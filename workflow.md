# LTS（LifeTech Select）運用・経営ワークフロー

本ドキュメントは、当アフィリエイトメディアの収益最大化戦略（PDCA）および、日々の記事作成手順（SOP）をまとめたものです。AIエージェントは必ずこのワークフローに従って業務を遂行してください。

---

## 第1部：収益最大化のためのPDCAサイクル（経営戦略）

当サイトは高単価（3万円以上）特化型メディアであるため、**CVR（成約率）の維持向上**と**検索流入の獲得**を至上命題とします。

### 【C】Check（データ解析） - 月2回実施
セキュリティおよび2段階認証の壁を越えるため、データの取得は**取締役（人間）がCSVをエクスポートしてAIに渡すフロー**とします。
1. **Google Search Console**: 検索パフォーマンス画面から「検索クエリ」「掲載順位」「CTR」のデータをCSV出力し、AIへ提示する。
2. **楽天アフィリエイト**: レポート画面から「クリック数」「CVR（成約率）」「発生報酬」のデータをCSV出力し、AIへ提示する。

### 【A】Action（サイト最適化） - 解析直後
データに基づき、以下の改修を実施する。
- **検索表示は多いがCTRが低い** → `articles.yaml` のSEOタイトル・メタ情報を改善。
- **クリックされない/成約しない** → `best_for`（ペルソナ）や `cost_performance`（投資回収ロジック）を書き換える。
- **在庫切れ** → `keyword` を後継機種に差し替えて再ビルド。

## 第2部：記事作成・更新SOP（戦術実行手順）

新しいカテゴリを追加する際や、既存記事を再構築する際は、必ず以下の手順（LTS絶対基準）を守ること。

### 1. 市場リサーチ（名機選定）
楽天APIに汎用キーワードを投げるのは**厳禁**。必ず事前にウェブ検索（`search_web`）を行い、読者が「指名買い」するレベルの真の名機（トップブランド）を3〜4機種に厳選する。

### 2. KBF（Key Buying Factor）の策定
汎用的な星評価は使用しない。そのカテゴリ特有の「ユーザーの本当の悩み」から、3つの評価軸を独自に定義する。

### 3. `articles.yaml` の高度な設定構築
- `min_price`: 安物（ノイズ）を弾くための最低価格を設定する。
- `rakuten_params.keyword`: 1で特定した**名機の型番・ブランド名**をカンマ区切りで指定する。
- `products_extra`: 各名機に対し、以下のLTS独自解析を**必ず全項目記述**する。
  - `best_for`（推奨ターゲット）
  - `scores`（KBFスコア）
  - `analysis_why`（解析チームの選定理由）
  - `pros`（支持されている点）
  - `critical_cons`（致命的な弱点）
  - `maintenance_reality`（メンテナンスの現実）
  - `cost_performance`（コスパの正体）

### 4. システム実行と品質検証（品質ゲート）
```powershell
python src\fetch_products.py
python src\build_site.py
```
`build_site.py` には取得製品数と解析テキスト数の不一致をブロックするゲートが存在する。ビルドエラーが出た場合は決して強行せず、`articles.yaml` の設定を修正すること。

### 5. デプロイ
ローカルでエラーなく生成されたことを確認後、Gitでコミット＆プッシュ（本番公開）を行う。

## 第3部：日次業務ルーティン（Operational Triggers）

このプロジェクトを稼働させるAIエージェントは、日次業務として必ず以下の2つのタスクを実行すること。

### ① 日次公開用ページの作成（計画公開）
- 定常的な高単価カテゴリ（`type: daily` / `category_id: home`等）の記事を作成する。
- 記事のストック（未来の日付 `release_date` が設定された完成済み記事）が存在しない場合、バックログから新たなテーマを選び、第2部のSOPに従ってLTS水準の記事を1本作成し、公開スケジュールに乗せること。

### ② SNSトレンドの即日公開（突発公開）
- `config/trend_sources.yaml` 等のソースをウェブ検索（`search_web`）で巡回し、海外のRedditやSNSで直近バズっているガジェット・トレンド商品がないか監視する。
- バズっている商品を発見した場合、計画公開とは**別枠**で、即座にその商品のLTS記事（`type: sns` / `category_id: trend`）を第2部のSOPに従って作成し、**`release_date` を当日（即日公開）に設定してデプロイ**すること。
